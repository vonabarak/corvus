# Cloud-Init Reference

Corvus uses cloud-init to provision VMs at first boot. When cloud-init is enabled on a VM, the daemon generates a [NoCloud](https://cloudinit.readthedocs.io/en/latest/reference/datasources/nocloud.html) ISO containing user-data, meta-data, and optionally network-config. The ISO is attached to the VM as a read-only IDE CD-ROM and cloud-init inside the guest reads it on startup.

## Enabling Cloud-Init

Cloud-init is controlled by the `cloudInit` flag on a VM:

| Method | How to enable |
|--------|---------------|
| `crv vm create` | `--cloud-init` flag |
| `crv vm edit` | `--cloud-init true` |
| Apply YAML | `cloudInit: true` on a VM entry |
| Template | `cloudInit: true` (inherited on instantiation) |

When a VM has SSH keys attached, cloud-init is required — attaching an SSH key to a VM with `cloudInit: false` is rejected.

In apply YAML, cloud-init auto-detection is available: omit the `cloudInit` field and it is automatically enabled when the VM has `sshKeys` defined.

## CLI Commands

```bash
crv cloud-init set <vm> [file.yml]    # Set config from YAML file or open $EDITOR
crv cloud-init edit <vm>              # Edit current config in $EDITOR
crv cloud-init show <vm>              # Show current config
crv cloud-init delete <vm>            # Delete custom config (revert to defaults)
crv cloud-init generate <vm>          # Regenerate the ISO manually
```

### Setting Configuration

`crv cloud-init set <vm> <file.yml>` reads a YAML file and uploads it. Without a file argument, `$EDITOR` (or `vi`) opens on a skeleton:

```yaml
injectSshKeys: true
# userData:
#   packages:
#     - qemu-guest-agent
#   runcmd:
#     - systemctl enable qemu-guest-agent
# networkConfig:
#   version: 2
#   ethernets:
#     eth0:
#       dhcp4: true
```

### Editing Configuration

`crv cloud-init edit <vm>` fetches the current config from the server, writes it to a temporary `.yml` file, and opens `$EDITOR`. If no custom config exists yet, the skeleton is used. On save, the file is parsed and uploaded. If saved without changes, no update is sent.

### Show and Delete

`crv cloud-init show <vm>` displays the current config (or "Using default cloud-init configuration" if none is set). With `--output json` or `--output yaml`, the raw `CloudInitInfo` structure is emitted.

`crv cloud-init delete <vm>` removes the custom config and regenerates the ISO with defaults.

## Configuration YAML

The cloud-init config YAML has three fields. The same format is used by the `crv cloud-init set` command, the `cloudInitConfig` section in templates, and the `cloudInitConfig` section in apply YAML VMs.

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `injectSshKeys` | bool | `true` | Merge SSH keys from the database into the first user's `ssh_authorized_keys`. |
| `userData` | string or YAML | | Custom user-data. Replaces the default cloud-config when set. |
| `networkConfig` | string or YAML | | Custom network-config (cloud-init v2 format). Not generated by default. |

### User-Data Format

The `userData` field accepts two formats:

**Structured YAML** (cloud-config): parsed as a YAML object, stored in the ISO with a `#cloud-config` header prepended by Corvus:

```yaml
userData:
  users:
    - name: deploy
      sudo: "ALL=(ALL) NOPASSWD:ALL"
      shell: /bin/bash
  packages:
    - nginx
    - postgresql
  runcmd:
    - systemctl enable nginx
```

**Raw script**: any text starting with `#` is used verbatim. This is how you supply scripts for cloud-init variants like cloudbase-init on Windows:

```yaml
userData: |
  #ps1_sysnative
  net user Administrator "Password123!" /y
  Install-WindowsFeature -Name RDS-RD-Server
  Start-Service TermService
```

Other script types (`#!/bin/bash`, `#cloud-boothook`, etc.) are also passed through as-is.

### SSH Key Injection

When `injectSshKeys` is `true` (the default), SSH keys attached to the VM in the database are merged into the user-data:

- If `userData` contains a `users` array, keys are appended to the first user's `ssh_authorized_keys`.
- If `userData` has no `users` section, a top-level `ssh_authorized_keys` key is added.
- If `userData` is a raw script (starts with `#`), injection is skipped — raw scripts cannot be merged.
- Existing keys in the `ssh_authorized_keys` array are preserved; new keys are appended.

Set `injectSshKeys: false` when the user-data is a raw script or when you manage SSH keys manually.

### Network Config

The optional `networkConfig` field provides a [cloud-init network config v2](https://cloudinit.readthedocs.io/en/latest/reference/network-config-format-v2.html) document. It is written as-is to the `network-config` file in the ISO. When omitted, no network-config file is included and the guest uses its default network configuration.

```yaml
networkConfig:
  version: 2
  ethernets:
    eth0:
      dhcp4: true
```

## Default Configuration

When no custom config is set, Corvus generates a default cloud-config with:

| Setting | Value |
|---------|-------|
| User | `corvus` |
| Password | `corvus` (plain text, non-expiring) |
| Shell | `/bin/sh` |
| Sudo | `ALL=(ALL) NOPASSWD:ALL` |
| SSH password auth | enabled |
| Root | disabled |
| Host keys | RSA + Ed25519 (generated if missing) |
| Packages | `qemu-guest-agent` |
| Run commands | Set permissions on `~/.ssh`, restart SSH daemon, enable and start `qemu-guest-agent` |

The default user-data also includes a `doas` rule (`permit nopass corvus`) for systems that use doas instead of sudo.

## ISO Generation

### When the ISO is Generated

The cloud-init ISO is generated or regenerated automatically:

| Event | Trigger |
|-------|---------|
| **VM start** | If `cloudInit` is enabled, the ISO is regenerated before the QEMU process starts. |
| **SSH key attach** | Attaching an SSH key to a cloud-init-enabled VM regenerates the ISO to include the new key. |
| **SSH key detach** | Detaching a key regenerates the ISO without it. |
| **`crv cloud-init set`** | After updating the config, the ISO is immediately regenerated. |
| **`crv cloud-init delete`** | After deleting custom config, the ISO is regenerated with defaults. |
| **`crv cloud-init generate`** | Explicit manual regeneration. |
| **Template instantiation** | If the template has `cloudInit: true`, the ISO is generated for the new VM. |

### ISO Contents

The NoCloud ISO (volume label `cidata`) contains:

| File | Source | Always present |
|------|--------|----------------|
| `user-data` | Custom config or default generation | yes |
| `meta-data` | `instance-id` + `local-hostname` (derived from VM name) | yes |
| `network-config` | Custom `networkConfig` field | only if configured |

The ISO is created using `genisoimage` (with `mkisofs` as fallback) with Joliet + Rock Ridge extensions.

### ISO Storage

The ISO is stored alongside the VM's disk images:

```
$HOME/VMs/<vm-name>/cloud-init.iso
```

It is registered as a disk image in the database (name: `<vm-name>-cloud-init`, format: `raw`) and attached as a read-only IDE CD-ROM drive with no caching.

## Templates and Apply Integration

### Templates

Templates support `cloudInitConfig` at the top level. When a VM is instantiated from a template, the cloud-init config is copied to the new VM and the ISO is generated as part of the instantiation process.

```yaml
name: webserver
cpuCount: 2
ramMb: 2048
cloudInit: true
guestAgent: true
cloudInitConfig:
  userData:
    packages:
      - nginx
    runcmd:
      - systemctl enable nginx
  injectSshKeys: true
drives:
  - diskImageName: ubuntu-base
    interface: virtio
    strategy: overlay
sshKeys:
  - name: deploy-key
```

See [templates.md](templates.md) for the full template reference.

### Apply YAML

VMs in apply configurations support the same `cloudInitConfig` section:

```yaml
vms:
  - name: web-server
    cpuCount: 2
    ramMb: 2048
    cloudInit: true
    cloudInitConfig:
      userData:
        packages:
          - nginx
      injectSshKeys: true
    sshKeys:
      - deploy-key
    drives:
      - disk: web-root
        interface: virtio
```

Auto-detection: if `cloudInit` is omitted and the VM has `sshKeys`, cloud-init is enabled automatically.

See [apply-configuration.md](apply-configuration.md) for the full apply reference.

## Windows Support (Cloudbase-Init)

Windows guests use [cloudbase-init](https://cloudbase-init.readthedocs.io/) instead of cloud-init. Corvus generates the same NoCloud ISO, but the user-data must be a PowerShell script (or other format cloudbase-init understands):

```yaml
injectSshKeys: false
userData: |
  #ps1_sysnative
  net user Administrator "corvus" /y
  Install-WindowsFeature -Name RDS-RD-Server
  Set-ItemProperty -Path 'HKLM:\System\CurrentControlSet\Control\Terminal Server' -Name fDenyTSConnections -Value 0
  Enable-NetFirewallRule -DisplayGroup "Remote Desktop"
  Set-Service -Name TermService -StartupType Automatic
  Start-Service TermService
```

**Requirements** for the Windows guest image:
- Cloudbase-init installed with `NoCloudConfigDriveService` enabled in its config (the ISO volume label `cidata` matches this service).
- `first_logon_behaviour=no` in cloudbase-init config to prevent password randomization after the script runs.

Set `injectSshKeys: false` — SSH key injection modifies YAML structure which is incompatible with raw PowerShell scripts.

## Troubleshooting

**Cloud-init doesn't run in the guest**:
- Verify cloud-init (or cloudbase-init) is installed in the guest image.
- Check that the ISO is attached: `crv vm show <vm>` should list a `cloud-init` CD-ROM drive.
- For Windows, ensure `NoCloudConfigDriveService` is in cloudbase-init's `metadata_services` config.

**SSH keys not working**:
- Verify `injectSshKeys: true` (the default) in the cloud-init config.
- Check that keys are attached to the VM: `crv ssh-key list-for-vm <vm>`.
- Raw script user-data (starting with `#`) skips key injection.

**Changes not taking effect**:
- Cloud-init typically only runs on first boot. To re-run, delete `/var/lib/cloud/instance` inside the guest and reboot, or use `cloud-init clean` if available.
- On Windows, cloudbase-init runs once per boot by default.

**ISO generation fails**:
- Requires `genisoimage` or `mkisofs` installed on the host.
- Check daemon logs for error messages.
